# Requirements Document

## Introduction

The agentic event orchestrator in `packages/agentic_event_orchestrator` currently has limited vendor visibility and no autonomous action capability. The `VendorDiscoveryAgent` can run a basic hybrid search and fetch a single vendor's details, but it cannot check availability for a specific date, compare multiple vendors side-by-side, list a vendor's services, or proactively surface recommendations when a user creates an event. The `BookingAgent` calls the backend without service-auth headers on some paths, and no tool exists to query vendor availability before initiating a booking.

This feature adds the missing `@function_tool` definitions, wires them into the correct agents, extends the `VendorDiscoveryAgent` to support all three search modes (keyword, semantic, hybrid), and gives the `OrchestratorAgent` the autonomous capability to compare vendors and proactively recommend them — all while preserving the mandatory user-confirmation gate before any booking is created.

## Glossary

- **Orchestrator**: The `packages/agentic_event_orchestrator` FastAPI service running on port 8000.
- **Backend**: The `packages/backend` FastAPI service running on port 5000.
- **VendorDiscoveryAgent**: The specialist agent responsible for vendor search, detail retrieval, availability checking, and comparison.
- **BookingAgent**: The specialist agent responsible for creating, viewing, and cancelling bookings.
- **OrchestratorAgent**: The master coordinator agent that runs multi-step workflows autonomously.
- **TriageAgent**: The sole entry point for all user interactions; routes to specialist agents.
- **function_tool**: An OpenAI Agents SDK decorator (`@function_tool`) that wraps an async Python function and exposes it to the LLM as a callable tool.
- **Service_Auth_Headers**: HMAC-SHA256 signed headers (`X-Service-Timestamp`, `X-Service-Signature`, `X-Service-Name`) generated by `service_auth.make_service_headers()` and required on every outbound HTTP call from the Orchestrator to the Backend.
- **Public_Vendor_API**: The unauthenticated Backend endpoints under `/api/v1/public_vendors` — supports keyword, semantic, and hybrid search, vendor profile retrieval, and autocomplete suggestions.
- **Vendor_API**: The authenticated Backend endpoints under `/api/v1/vendors` — includes availability slots, services, and booking management.
- **Availability_Slot**: A record in `vendor_availability` representing a vendor's available or blocked date range for a specific service, with fields: `id`, `vendor_id`, `service_id`, `start_date`, `end_date`, `is_available`.
- **Vendor_Comparison**: A structured side-by-side summary of two or more vendors across rating, price range, capacity, and availability for a given date.
- **Proactive_Recommendation**: An unsolicited vendor suggestion surfaced by the OrchestratorAgent immediately after an event is created, without the user explicitly asking for vendors.
- **Search_Mode**: One of `keyword`, `semantic`, or `hybrid` — controls which Backend search algorithm is used.
- **Confirmation_Gate**: The mandatory step where the BookingAgent presents a summary table and requires the user to reply "confirm" before `create_booking_request` is called.

## Requirements

### Requirement 1: Vendor Availability Checking Tool

**User Story:** As a user planning an event, I want the AI assistant to tell me whether a specific vendor is available on my event date, so that I only consider vendors who can actually serve me.

#### Acceptance Criteria

1. THE Orchestrator SHALL expose a `check_vendor_availability` `function_tool` that accepts `vendor_id` (string), `event_date` (ISO-8601 date string `YYYY-MM-DD`), and an optional `service_id` (string, default `None`).
2. WHEN `check_vendor_availability` is called, THE Orchestrator SHALL send a GET request to `{BACKEND_API_URL}/api/v1/vendors/{vendor_id}/availability` with `start_date` and `end_date` query parameters both set to `event_date`; IF `service_id` is provided, THE Orchestrator SHALL also include it as a `service_id` query parameter; THE request SHALL include Service_Auth_Headers merged into the outgoing request headers.
3. WHEN the Backend returns a 200 response, THE `check_vendor_availability` tool SHALL return a JSON string containing: `vendor_id` (string), `available` (boolean, `true` when the returned slots list is non-empty), and `slots` (list of Availability_Slot objects, each with `id`, `vendor_id`, `service_id`, `start_date`, `end_date`, `is_available`).
4. IF the Backend returns a non-200 status code, THEN THE `check_vendor_availability` tool SHALL return a JSON string with `available` set to `false`, `vendor_id` set to the requested vendor ID, `slots` set to an empty list, and an `error` field containing the HTTP status code and the `error.message` from the response body if present, otherwise the response text.
5. IF a network exception occurs during the request, THEN THE `check_vendor_availability` tool SHALL catch the exception and return a JSON string with `available` set to `false`, `vendor_id` set to the requested vendor ID, `slots` set to an empty list, and an `error` field containing the exception class name and message.
6. THE `check_vendor_availability` tool SHALL be attached to the VendorDiscoveryAgent's `tools` list.

### Requirement 2: Vendor Services Listing Tool

**User Story:** As a user, I want the AI assistant to show me the specific services a vendor offers (with prices and capacity), so that I can choose the right service before booking.

#### Acceptance Criteria

1. THE Orchestrator SHALL expose a `get_vendor_services` `function_tool` that accepts `vendor_id` (string).
2. WHEN `get_vendor_services` is called and a response is received from the Backend, THE Orchestrator SHALL extract the `services` field from the response data of a GET request to `{BACKEND_API_URL}/api/v1/public_vendors/{vendor_id}`, including Service_Auth_Headers merged into the outgoing request headers.
3. WHEN the Backend returns a 200 response containing a vendor profile, THE `get_vendor_services` tool SHALL return a JSON string with a `services` list containing only entries where `is_active` is `true`, where each entry includes `id`, `name`, `price_min`, `price_max`, and `capacity`.
4. IF the vendor has no active services or the `services` field is absent, THEN THE `get_vendor_services` tool SHALL return a JSON string with an empty `services` list and a `vendor_id` field set to the requested vendor ID.
5. IF the Backend returns a non-200 status code OR the Backend is unreachable (network exception), THEN THE `get_vendor_services` tool SHALL return a JSON string with an `error` field containing a message indicating the failure reason, and an empty `services` list.
6. THE `get_vendor_services` tool SHALL be attached to both the VendorDiscoveryAgent's and BookingAgent's `tools` lists.
7. IF `vendor_id` is an empty string, THEN THE `get_vendor_services` tool SHALL return a JSON string with an `error` field stating "vendor_id must not be empty" and an empty `services` list, without making any HTTP request.

### Requirement 3: Multi-Vendor Comparison Tool

**User Story:** As a user, I want the AI assistant to compare multiple vendors for me on rating, price, and availability, so that I can make an informed choice without manually looking up each vendor.

#### Acceptance Criteria

1. THE Orchestrator SHALL expose a `compare_vendors` `function_tool` that accepts `vendor_ids` (list of strings, minimum 2, maximum 10), `event_date` (ISO-8601 date string), and an optional `criteria` parameter (list of strings, defaulting to `["rating", "price", "availability"]`).
2. WHEN `compare_vendors` is called, THE Orchestrator SHALL fetch the vendor profile for each `vendor_id` by calling `get_vendor_details` and the availability for each by calling `check_vendor_availability` for the given `event_date`; these fetches MAY be performed concurrently.
3. WHEN all vendor data has been fetched, THE `compare_vendors` tool SHALL return a JSON string containing a `comparison` list where each entry includes `vendor_id`, `business_name`, `rating`, `price_min`, `price_max`, `available` (boolean), and `city`; the list SHALL be ordered by `rating` descending, with ties broken by `business_name` ascending.
4. IF fewer than two `vendor_ids` are provided, THEN THE `compare_vendors` tool SHALL return a JSON string with an `error` field stating "At least two vendor IDs are required for comparison" and no `comparison` field.
5. IF `vendor_ids` contains more than 10 entries or contains any empty-string entries, THEN THE `compare_vendors` tool SHALL return a JSON string with an `error` field describing the validation failure and no `comparison` field, without making any HTTP requests.
6. IF fetching details for a specific vendor fails, THEN THE `compare_vendors` tool SHALL include that vendor in the `comparison` list with `vendor_id` populated and all other fields (`business_name`, `rating`, `price_min`, `price_max`, `available`, `city`) set to `null`, rather than omitting the vendor entirely.
7. THE `compare_vendors` tool SHALL be attached to the VendorDiscoveryAgent's and OrchestratorAgent's `tools` lists.
8. IF `event_date` is a date in the past (before the current UTC date), THEN THE `compare_vendors` tool SHALL return a JSON string with an `error` field stating "event_date must be a future date" and no `comparison` field.

### Requirement 4: Extended Vendor Search with Configurable Mode

**User Story:** As a user, I want the AI assistant to find vendors using the most relevant search strategy (keyword, semantic, or hybrid), so that I get accurate results whether I describe my needs in plain language or use specific terms.

#### Acceptance Criteria

1. THE `search_vendors` `function_tool` SHALL accept a `mode` parameter of type string with allowed values `"keyword"`, `"semantic"`, and `"hybrid"`, defaulting to `"hybrid"`.
2. WHEN `search_vendors` is called, THE Orchestrator SHALL pass the `mode` parameter as a query parameter to `{BACKEND_API_URL}/api/v1/public_vendors/search`, including Service_Auth_Headers merged into the outgoing request headers.
3. WHEN the Backend returns a 200 response, THE `search_vendors` tool SHALL return a JSON string with a `vendors` list and a `total` count; IF a `search_mode` field is present on individual result entries in the backend response, THE tool SHALL preserve it on each entry.
4. IF `mode` is set to `"semantic"` and the Backend returns a 503 with error code `AI_EMBEDDING_UNAVAILABLE`, THEN THE `search_vendors` tool SHALL return a JSON string with an `error` field containing the message "Semantic search is temporarily unavailable. Please try keyword or hybrid search." and an empty `vendors` list.
5. IF the Backend returns any non-200 response not covered by criterion 4, THEN THE `search_vendors` tool SHALL return a JSON string with an `error` field containing the HTTP status code and response message, and an empty `vendors` list.
6. THE VENDOR_DISCOVERY_INSTRUCTIONS system prompt SHALL instruct the VendorDiscoveryAgent to: use `mode="semantic"` when the user's query contains descriptive or qualitative phrasing (e.g., "elegant", "affordable", "good for outdoor events"); use `mode="keyword"` when the user provides a recognized category label (e.g., "catering", "photography") or a specific vendor name; use `mode="hybrid"` (default) in all other cases.

### Requirement 5: Proactive Vendor Recommendations After Event Creation

**User Story:** As a user, I want the AI assistant to automatically suggest relevant vendors right after I create an event, so that I can start booking without having to ask separately.

#### Acceptance Criteria

1. WHEN the EventPlannerAgent successfully creates an event, THE EventPlannerAgent SHALL ask the user "Would you like me to find vendors for this event?" before handing off to the VendorDiscoveryAgent.
2. IF the user responds affirmatively (e.g., "yes", "sure", "please", "ok") or does not explicitly decline within the same conversation turn, THEN THE EventPlannerAgent SHALL hand off to the VendorDiscoveryAgent with the event's `event_type`, `city`, and `budget` as context.
3. IF the user explicitly declines (e.g., "no", "not now", "skip"), THEN THE EventPlannerAgent SHALL acknowledge the response and end the turn without handing off to the VendorDiscoveryAgent.
4. WHEN the VendorDiscoveryAgent receives a handoff from EventPlannerAgent with event context, THE VendorDiscoveryAgent SHALL call `search_vendors` without requesting additional information from the user in the same agent turn.
5. WHEN the VendorDiscoveryAgent presents proactive recommendations, THE VendorDiscoveryAgent SHALL display the top 3 vendors sorted by `rating` descending, with one vendor per line in the format: `{business_name} — {category} — PKR {price_min}–{price_max} — ⭐ {rating}`, with no prose between entries.
6. IF `search_vendors` returns zero results, THEN THE VendorDiscoveryAgent SHALL inform the user that no vendors were found for their event type and location, and suggest adjusting the city or budget.
7. IF the handoff context is missing `event_type` or `city`, THEN THE VendorDiscoveryAgent SHALL ask the user for the missing field(s) before calling `search_vendors`, asking one field at a time.

### Requirement 6: Autonomous Multi-Step Vendor Selection Workflow

**User Story:** As a user, I want the AI assistant to autonomously handle the full workflow of finding vendors, checking their availability, comparing them, and presenting a recommendation — without me having to drive each step manually.

#### Acceptance Criteria

1. WHEN the OrchestratorAgent is handling a multi-step vendor selection request, THE OrchestratorAgent SHALL autonomously call `search_vendors`, then call `check_vendor_availability` for up to 3 of the top-rated results, then call `compare_vendors` on those vendors, and present a ranked recommendation — all within a single conversation turn without asking the user for intermediate confirmations between tool calls.
2. THE ORCHESTRATOR_INSTRUCTIONS system prompt SHALL instruct the OrchestratorAgent to execute the vendor selection workflow (search → availability check → compare → recommend) without pausing for user input between tool calls.
3. WHEN the OrchestratorAgent presents its final recommendation, THE OrchestratorAgent SHALL include: (a) the names of all vendors checked, (b) which vendors are available on the requested date, and (c) the top recommended vendor with a rationale citing at least one of: rating, price range, or availability.
4. WHEN the OrchestratorAgent has identified the top recommendation and the user confirms they want to proceed, THE OrchestratorAgent SHALL hand off to the BookingAgent to complete the Confirmation_Gate flow; THE OrchestratorAgent SHALL NOT call `create_booking_request` directly.
5. IF all vendors returned by `search_vendors` are unavailable on the requested date, THEN THE OrchestratorAgent SHALL inform the user and offer to search for up to 3 alternative dates within a 30-day window from the original date, or offer to search in a different city.
6. IF `search_vendors` returns zero results, THEN THE OrchestratorAgent SHALL inform the user that no vendors were found and suggest adjusting the event type, city, or budget, without proceeding to availability checking or comparison.

### Requirement 7: Service Authentication on All Outbound Tool Calls

**User Story:** As a platform operator, I want every HTTP call from the orchestrator to the backend to carry a valid service-auth signature, so that the backend can reject unauthenticated callers and prevent abuse.

#### Acceptance Criteria

1. THE Orchestrator SHALL use `service_auth.make_service_headers(method, path)` to generate Service_Auth_Headers for every outbound HTTP request made by any `function_tool`.
2. WHEN a `function_tool` constructs an `httpx.AsyncClient` request, THE tool SHALL merge the Service_Auth_Headers returned by `make_service_headers` into the outgoing request headers (not replacing other headers).
3. THE `check_vendor_availability` tool SHALL include Service_Auth_Headers on its GET request to `/api/v1/vendors/{vendor_id}/availability`.
4. THE `get_vendor_services` tool SHALL include Service_Auth_Headers on its GET request to `/api/v1/public_vendors/{vendor_id}`.
5. THE `search_vendors` tool SHALL include Service_Auth_Headers on its GET request to `/api/v1/public_vendors/search`.
6. THE `get_vendor_details` tool SHALL include Service_Auth_Headers on its GET request to `/api/v1/public_vendors/{vendor_id}`.
7. IF `SERVICE_SECRET` is not set in the environment OR is set to an empty string, THEN THE Orchestrator SHALL log a warning at startup indicating that the service secret is missing or empty and the development fallback is in use, and SHALL use the development fallback secret `"dev-insecure-secret"` as defined in `service_auth.py`.
8. IF the Backend returns a 401 or 403 response to any tool call, THEN THE tool SHALL return a JSON string with an `error` field containing the message "Service authentication failed. Check SERVICE_SECRET configuration." and, where applicable, an empty result list or `available: false`.

---

### Requirement 8: Graceful Degradation on Backend Unavailability

**User Story:** As a user, I want the AI assistant to give me a clear, friendly error message when vendor data is temporarily unavailable, rather than crashing or returning a confusing technical error.

#### Acceptance Criteria

1. IF any `function_tool` encounters a network exception (e.g., `httpx.ConnectError`, `httpx.TimeoutException`) when calling the Backend, THEN THE tool SHALL catch the exception and return a JSON string with an `error` field that does not contain raw stack traces, exception class names, or internal hostnames, and SHALL include an empty result list where the tool normally returns a list.
2. IF the Backend returns an HTTP 500, 502, 503, or 504 response to any tool call, THEN THE tool SHALL return a JSON string with an `error` field indicating the service is temporarily unavailable, and SHALL include an empty result list where the tool normally returns a list.
3. IF a tool returns a JSON string containing an `error` field, THEN THE VendorDiscoveryAgent SHALL relay the error to the user in plain language without including raw HTTP status codes, exception class names, or internal path information.
4. THE `search_vendors`, `get_vendor_details`, `check_vendor_availability`, `get_vendor_services`, and `compare_vendors` tools SHALL each guarantee that a valid JSON string is always returned to the caller — a raised exception SHALL never propagate out of any of these tools.

---

### Requirement 9: Booking Confirmation Gate Enforcement

**User Story:** As a user, I want the AI assistant to always show me a clear summary and ask for my explicit confirmation before creating a booking, so that I never accidentally book a vendor.

#### Acceptance Criteria

1. BEFORE calling `create_booking_request`, THE BookingAgent SHALL present a markdown summary table containing `vendor_name`, `service_name`, `event_date`, `guest_count`, and `price_range` formatted as "PKR {price_min}–{price_max}".
2. WHEN the summary is presented, THE BookingAgent SHALL ask the user to reply "confirm" to proceed or "cancel" to abort, and SHALL NOT call `create_booking_request` until the user replies "confirm" (case-insensitive).
3. IF the user replies with any message other than "confirm" (case-insensitive), THEN THE BookingAgent SHALL treat the response as a cancellation, inform the user that the booking was not created, and offer to restart the booking process.
4. THE `create_booking_request` `function_tool` in `pipeline/booking.py` SHALL retain its `tool_input_guardrails=[tool_injection_guard]` and `tool_output_guardrails=[tool_pii_redact]` decorators.
5. THE BOOKING_INSTRUCTIONS system prompt SHALL explicitly state that calling `create_booking_request` without prior user confirmation is a critical violation; IF the agent attempts to call `create_booking_request` without having presented the summary and received "confirm", THE agent SHALL respond with an error message indicating the confirmation gate was skipped and SHALL NOT proceed with the booking.
6. WHEN `create_booking_request` returns a successful response, THE BookingAgent SHALL present a confirmation message to the user that includes the booking reference ID.
7. IF `create_booking_request` returns an error response, THEN THE BookingAgent SHALL inform the user of the failure with the reason provided, and SHALL NOT retry the booking without re-presenting the summary table and receiving a new "confirm" from the user.
